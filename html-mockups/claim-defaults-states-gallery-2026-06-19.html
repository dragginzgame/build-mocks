Toko · Consistency reference

Collection — Claim defaults.

The collection-level Claim defaults page — the default claim cost and rule for each rarity tier; new vendors drawing from the collection start here, then edit per vendor. This single reference combines three views: Part A — the page itself (set & unset); Part B — how it behaves across the collection lifecycle (Draft → Review → Live); Part C — how it adapts to each rarity mode (Uniform / Tiered / Weighted). Four concepts per tier: Cost (required — free / ICP / ICRC-1), Requirements, Restrictions and Rewards. Collection sub-page → auto-saves (no Save/Discard). Promo/redemption codes are project-level, not here.

Part A note (not UI) — why codes aren't hereCollection claim defaults describe the durable, rarity-based way a collection's tiers are sold (cost, neuron stake, whitelist, restrictions) — sensible to reuse across every vendor drawn from the collection. Promo and redemption codes are project-level code sets (campaign access, often single-use, tied to a particular drop/partnership) attached to a vendor or source — so one code can unlock a multi-collection vendor without duplicating pools per collection. They don't belong in a collection-wide default. A future addition (backend currently rejects these requirement types). Hold-token gating is also deferred.

Part C note (not UI)Four concepts per tier: Cost (what you pay — required, one of free/ICP/ICRC-1), Requirements (what you must prove/have — optional, AND-combined; none = anyone can claim), Restrictions (how often/much — optional; "max per human" coming soon), Rewards (extra payout — optional; ICP/ICRC-1 coming soon). Invalid inputs are prevented at entry, not deferred to go-Live. Only Live whitelists appear in the picker (a non-Live list can't be selected); a custom ICRC-1 cost ledger or SNS governance canister must resolve before it can be saved (frame 08), so an unresolved token/canister can never be attached. That leaves a single thing claim defaults gate — a tier with no selected cost — which is required before Review (and the whole thing locks at Live); it's surfaced on the collection's Draft→Review status checklist (see collection-states-gallery and claim-defaults-stage-states-gallery), not on this sub-page. Empty requirements/restrictions/rewards are never errors. Promo/redemption codes are project-level (future), not collection claim defaults.
