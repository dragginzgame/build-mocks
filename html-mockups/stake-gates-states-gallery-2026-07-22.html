Stake gates — NNS + SNS · V1 enablement

Change gallery for enabling the two stake gates in V1 (Donna, 2026-07-22 — prompted by Remco: the backend NNS/SNS claim checks are already working). "Has a stake" stops being a disabled teaser and becomes authorable wherever claim requirements are edited — the vendor tier editor and the collection-defaults rule editor — and visible wherever requirements are shown. Holds tokens from a collection stays front-end V2 and fail-closed (R75); Restrictions and Rewards are untouched. Requirement semantics are unchanged: types combine with AND; whitelists remain one union gate (ruling 2026-07-22); a tier may carry one NNS and one SNS gate, both binding.

Change spec: Design Documentation/stake-gates-v1-enablement.md. Base galleries this modifies: Vendor creation (frame 07/A26) · Collection defaults (A2/A2b) · Vendor claim (frames 01/02/05).
01Requirement editor — stake gates authorableThe tier requirement editor (vendor creation frame 07; the collection-defaults rule editor A2b gets the identical change) with the two stake accordions active instead of "Coming soon". Each gate: NNS = an ICP neuron; SNS = a governance canister that must resolve before Save enables. Both carry optional minimum_stake (the staked token — ICP for NNS, the SNS's native token otherwise) and optional minimum_dissolve_delay (dissolve delay, not neuron age — ruled 2026-07-03); both empty = any active stake. Within a gate the two fields AND together. One NNS + one SNS max per tier — a second SNS requirement is rejected at save (existing checklist rule). Token-hold stays a disabled teaser.
